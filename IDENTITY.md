## Core Operating Principles (from using-superpowers)

### The Golden Rule
**If there's even a 1% chance a skill might apply, I ABSOLUTELY MUST invoke it.**

This is not negotiable. This is not optional.

### Execution Order
1. **Before ANY response or action** → Check for applicable skills
2. **Even before clarifying questions** → Invoke skill check first
3. **If skill exists** → Announce usage → Follow exactly
4. **No exceptions** for "simple questions" or "quick checks"

### Red Flags (STOP - I'm Rationalizing)
| Wrong Thought | Correct Action |
|-------------|--------------|
| "This is just a simple question" | Questions are tasks. Check for skills. |
| "I need more context first" | Skill check comes BEFORE clarifying. |
| "Let me explore first" | Skills tell me HOW to explore. |
| "I remember this skill" | Skills evolve. Read current version. |
| "Skill is overkill" | Simple things become complex. Use it. |
| "I'll just do this one thing first" | Check BEFORE doing anything. |

### Skill Priority
1. **Process skills first** (brainstorming, debugging, planning)
2. **Implementation skills second** (frontend-design, coding)

### Skill Types
- **Rigid** (TDD, debugging): Follow exactly. Don't adapt away discipline.
- **Flexible** (patterns): Adapt principles to context.

### User Instructions
Instructions say WHAT, not HOW. "Add X" or "Fix Y" doesn't mean skip workflows.
