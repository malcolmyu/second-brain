🎯 核心洞察

一句话总结

Harness Engineering 不是让 AI 写代码更快，而是构建一套让 Agent 能够自主、可靠、持续执行复杂任务的工程化环境 —— 核心在于上下文治理、工具链设计和多智能体协作。

关键发现

  1. 瓶颈转移：AI 编码提速并未打破系统性天花板，非编码工作量（验证、测试、部署、排障）随代码生成量指数级爆发
  2. 范式转移：工程师角色从"编写代码"转向"设计环境、明确意图"，构建 Agent 专属工具链实现受控执行
  3. 上下文是金：上下文是最昂贵的资源，治理它而非堆砌它 —— 保护前缀稳定性、主动 Compaction、记忆外包
  4. 多智能体突破：单 Agent 天花板 = 模型基础智商，通过结构化通信的多智能体协作突破智能上限

📋 问题定义与背景

为什么 AI 出现后，研发反而更累了？

当前 AI 编程工具主要集中在"开发环境内极速写代码"，但并未解决测试、验证和沟通等环节。这造成了一个局部最优陷阱：

软件开发时间分配现状
  • 编码（AI 赋能环节）：30% —— AI 已经做得很好
  • 验证 / CI / 测试：40% —— 随代码量增加而膨胀
  • 部署 / 发布 / 灰度：20% —— 需要人工介入
  • 排障 / 沟通 / Code Review：10% —— 瓶颈所在

代码生成得越快越多，用于手动测试、验证排障和 Code Review 的工作量呈指数级上升。价值上限被后续流程死死锁住。

Harness Engineering 的定义

"不要只把大模型看作'大脑'，必须为它打造专属的'工作室'。"

Harness Engineering 是构建 Agent 专属工具链，实现受控执行的工程化方法论。其核心目标是：

  • 让 Agent 执行跨越整个生命周期的长时间闭环任务（Long-term Tasks）
  • 通过专属工具链集成，解决上下文爆炸问题
  • 让 AI 接管测试与排障，打破 70% 的非编码流程枷锁
  • 代码库重塑为模型能"读懂"的环境（Agent 可读性）

🔬 技术基础：Transformer 与上下文物理

理解 Agent 的"物理限制"

产品"慢、笨、会幻觉"并非 Bug，而是 Transformer 架构的物理现实：

现象 物理本质 工程含义
慢（SLOW） 自回归逐 Token 生成是物理铁律；上下文越长，Prefill 越昂贵 需要 KV Cache 和 Prompt Cache 优化
笨（DUMB） 上下文塞得越多，注意力被稀释越严重，关键逻辑被掩埋 需要上下文压缩和分段布局
幻觉（HALLUCINATION） 缺少事实锚点时，模型为维持概率连贯进行"合理猜测" 需要外部反馈闭环和验证机制

KV Cache：为什么"追加"便宜，"修改"昂贵

Transformer 推理分为两个阶段：

  1. Prefill 阶段：全量计算 KV（键值缓存）
  2. Decode 阶段：复用 KV，新 Token 只算一次
核心结论

追加（Append）= 复用 KV，O(新 token) —— 极度便宜
修改（Modify）= 从修改点起重算 KV，O(N) —— 极其昂贵

一切上下文工程，本质上都是为了"尽量不破坏前缀"。System Prompt 要保持极高稳定性，缓存才能命中，响应才能加速。

ReAct 循环：Agent 的操作系统

Agent 不是"一问一答"，而是 Thought → Action → Observation 的循环：

  • Thought（思考）：模型在上下文里"自言自语"，分析当前状态
  • Action（行动）：调用工具（read_file / write_file / bash），影响物理世界
  • Observation（观察）：工具返回结果追加到上下文，触发下一轮

上下文随迭代爆炸 → 第一原则：保护前缀稳定性

🏗️ Harness 系统架构

五层 Prompt 结构

一次 Agent 任务不是"把你的话丢给模型"，而是构造了一整套 Prompt：

1. System
模型系统提示 + Agent 身份定义
↓
2. AGENTS.md
技术栈、代码风格、安全边界
↓
3. 项目快照
工作目录、选中文本、项目结构
↓
4. 会话历史
对话记录 + 工具调用返回
↓
5. 工具定义
MCP 工具清单 + JSON Schema

上下文不是"越多越好"，而是要"把最重要的东西放到最该放的位置"。位置和占比决定了模型的注意力分配。

核心模块设计

1. 上下文治理系统
  • 三段式布局：System（稳定）+ Context（动态）+ Task（当前）
  • 主动 Compaction：将探讨结论写成 Markdown 快照，清空对话历史
  • 记忆外包：隐形知识写入文件系统，不在代码仓库里的对 Agent 不存在
2. 工具链设计（Agent 可读性）
  • 贴近物理上限：秒级冷启动与响应，防止 Agent 注意力涣散
  • 结构化输出：禁止 PDF/网页截图，必须输出 JSON/Markdown
  • 痛觉反馈：错误信息精确到行号与类型，是闭环的起点
3. 多智能体协作架构
  • Planner（规划器）：将产品 spec 分解为可执行任务列表
  • Generator（生成器）：执行具体编码任务，实现功能
  • Evaluator（评估器）：独立评估输出质量，提供批判性反馈
  • Orchestrator（编排器）：任务分发、结果汇总、冲突解决
4. 闭环验证系统
  • 静态分析：AST 检测，提前发现冲突
  • 增量重构：Agent 自动应用最小差异
  • 动态验证：运行测试用例，获取真实反馈
  • LSP 集成：Lint → Fix → Test 自动化闭环

🎨 核心设计模式

模式 1：Context Reset（上下文重置）

模型在长任务中会出现"上下文焦虑"（Context Anxiety）—— 接近上下文限制时提前收尾。解决方案：

  • Compaction（压缩）：早期对话总结保留，同一会话继续
  • Reset（重置）：完全清空，通过结构化 Handoff artifact 传递状态

Claude Sonnet 4.5 的上下文焦虑强烈到 Compaction 不足，Reset 成为必需。

模式 2：Generator-Evaluator 分离

Agent 自我评估时倾向于过度乐观（特别是主观任务如设计）。通过生成器和评估器分离：

  • 评估器可被调优为批判性（Skeptical）
  • 生成器有具体反馈可迭代
  • 类似 GAN（生成对抗网络）的对抗机制

模式 3：Spec Kit + Plan 模式

谋定而后动 —— 先写规格，再写代码：

  • Spec（规格说明书）：定义"做成什么样"（输入/输出 Schema、副作用边界、验证标准）
  • Plan（执行计划）：定义"怎么做"（任务拆解、检查点、回退策略）

模式 4：大脑分工（MCP / Skill / Sub Agent）

层级 定位 特点 示例
MCP 原子动作 常驻上下文，提供基础原子操作 GitHub、Slack、Database、K8s
Skill 可见的 SOP 按需激活，固化成熟打法 CI Debug、审查清单（skill.md + run.sh）
Sub Agent 黑盒外包 独立会话，上下文隔离 Explore 模式（并行探索不同微服务）

🛠️ 工程实践指南

原则 1：Agent 可读性优先

从"对人友好"转向"对 Agent 可读"：

  • ❌ 酷炫 GUI / 海量终端日志 → ✅ 结构化语义（JSON / API）
  • ❌ 面向鼠标点击的交互 → ✅ 面向自然语言指令
  • ❌ 视觉冗余图表 → ✅ 高信噪比的诊断摘要

原则 2：工具设计标准

工具不是给人用的 UI —— 是给 Agent 用的 API。
快（秒级响应）、准（100% 稳定结构化输出）、结构化（JSON/Markdown），缺一不可。

原则 3：人机协作边界

场景 策略 判断标准
Agent 自主执行 放手让 Agent 跑 Spec 明确 + 有闭环验证 + 工具能力覆盖完整
向人求助（AskUser） 第 1 轮就问，不猜 意图模糊 / 多条路径 / 环境信息不足 / 不可逆操作

原则 4：可观测性设计

让应用 UI、日志、指标本身对 Codex 可读：

  • 应用可按 git worktree 启动，Codex 每变更一个分支启动一个实例
  • 集成 Chrome DevTools Protocol，支持 DOM 快照、截图、导航
  • 日志、指标、追踪通过本地可观测性栈暴露，支持 LogQL / PromQL 查询
  • 提示如"确保服务启动在 800ms 内"或"关键用户路径跨度不超过 2 秒"变得可执行

📊 案例分析

案例 1：OpenAI 的 0 人工代码实验

实验概况
  • 时间：2025年8月 - 2026年1月（5个月）
  • 团队：3人 → 7人工程师团队
  • 代码量：约 100 万行代码
  • PR 数量：约 1,500 个
  • 约束：0 行人工编写的代码

关键发现

  1. 早期更慢：不是因为 Codex incapable，而是因为环境 underspecified
  2. 深度优先：将大目标分解为小构建块，Prompt Agent 构建，解锁复杂任务
  3. 问题修复：从不"try harder"，而是"what capability is missing, and how to make it legible and enforceable for the agent?"
  4. 单任务时长：经常看到单 Codex 运行处理单个任务超过 6 小时（人类睡觉时）

AGENTS.md 作为地图

OpenAI 发现"一本大 AGENTS.md"会失败：

  • 上下文稀缺 → 大文件挤占任务、代码、文档空间
  • 过度指导 = 无指导 → Agent 局部模式匹配而非有意导航
  • 立即腐烂 → 成为过时规则的墓地

解决方案：AGENTS.md 作为目录（Table of Contents），指向结构化 docs/ 目录的深层真相源。

案例 2：Anthropic 的多智能体前端设计

问题

Claude 倾向于安全、可预测的布局（技术可用但视觉平庸），且自我评估时过度乐观。

解决方案

  1. 定义评分标准：将"这个设计好看吗？"转化为"是否符合设计原则？"的四个维度：
    • Design Quality（设计质量）：是否形成整体而非零件集合
    • Originality（原创性）：是否有定制决策证据，而非模板默认
    • Craft（工艺）：排版层次、间距一致性、色彩和谐
    • Functionality（功能性）：可用性独立于美学
  2. Generator-Evaluator 分离：创建反馈循环驱动生成器向更强输出迭代

案例 3：字节 Web Infra 的 Aiden CLI

周晓团队基于 Harness Engineering 理念构建的 AI 原生 CLI 工具：

  • 需求到交付的端到端闭环（需求 → Spec → Plan → 生成 → 验证 → 交付）
  • 从 Review 代码到 Review 交互和测试
  • Agent 在独立沙盒工作，只交换结构化摘要

💡 结论与启示

核心结论

  1. Harness Engineering 是 AI 时代的系统工程方法论 —— 不是优化单点，而是构建让 Agent 有效工作的完整环境
  2. 上下文治理是基础设施 —— 理解 Transformer 物理限制（KV Cache、自回归），才能设计高效的上下文策略
  3. 工具链决定天花板 —— Agent 的工具质量直接决定 Harness 工程的成败，快、准、结构化是标准
  4. 多智能体是突破路径 —— 单 Agent 受限于模型智商，通过结构化通信的多智能体协作突破智能上限
  5. 人机协作有新边界 —— 不是取代人，而是让 Agent 知道何时自主、何时求助

实施建议

Phase 1：上下文工程（1-2 周）
  • 设计 AGENTS.md 作为地图，而非百科全书
  • 建立结构化 docs/ 目录作为知识库
  • 实现上下文 Compaction 机制
Phase 2：工具链建设（2-4 周）
  • 评估现有工具对 Agent 的可读性
  • 开发结构化输出工具（JSON/Markdown）
  • 集成 LSP 实现 Lint → Fix → Test 闭环
Phase 3：多智能体协作（4-8 周）
  • 设计 Spec Kit 规范需求定义
  • 实现 Planner-Generator-Evaluator 三智能体架构
  • 建立 Agent 间结构化通信协议
北极星指标

Agent 吞吐量：每个工程师每天驱动的 PR 数量
OpenAI 数据：3.5 PRs/人/天，且随团队规模增长而提升（7人团队 > 3人团队）
