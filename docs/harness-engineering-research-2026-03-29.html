Harness Engineering - 深度调研报告

核心洞察（5 条）

  1. Harness 是"临时脚手架"，其存在本身就是模型不足的证明
    每个组件都是对"模型在这里做不到什么"的假设编码，随着 Opus 4.5→4.6 的进化，Sprint 结构、上下文重置等组件被直接砍掉。

  2. Generator 与 Evaluator 必须分离——对抗式评估是质量保证的核心
    AI 有"结构性自恋"，自我评价总是偏高。分离后，Evaluator 能发现开发者自己看不到的问题（如用户消息气泡对齐方式反了）。

  3. 文件通信优于 RPC/消息队列——文本是 LLM 的自然语言
    三 Agent 之间没有复杂通信协议，就是读写文件。文本文件天然中立、可审计、可版本控制。

  4. 成本与质量的非线性关系：$9/20分钟 vs $200/6 小时
    单 Agent 得到的是"界面看起来差不多但核心功能坏掉的壳子"，三 Agent 得到的是"有精灵动画、关卡编辑、音效支持的完整创作工具"。

  5. 工程师角色正在从"代码搬运工"转向"工程规则设计师"
    AI 抹平语法熟练度，放大工程意识价值。未来的高级开发是设计让代码被正确写出来的系统。

背景与问题定义

核心问题

当 AI 处理长时间、复杂任务时，会出现哪些系统性失败？

Anthropic 识别出两大死穴： 1. 一口吃成胖子：给 AI 一句"做个 Web 应用"，它会尝试一次性把所有功能怼出来，写到一半 context 用完，下一轮换 Agent 接手时对前任工作一无所知 2. 过早宣布胜利：搞了几个功能后，AI 审视当前状态，得出结论"看起来差不多了"，然后收工，但核心逻辑根本就是空壳

为什么现在重要？

  • OpenAI 3 个工程师 5 个月用 Codex 产出百万行代码
  • Stripe 每周通过 Minions Agent 合并 1000+ PR
  • Cursor 估值 500 亿美元
  • 但同样模型，放到不同系统里，表现差得非常大

行业共识：Agent 的问题越来越不是"模型够不够聪明"，而是"环境够不够好"。

多维度分析

一、理论维度：从 Prompt 到 Harness 的演进

阶段 核心问题 类比 关键转变
Prompt Engineering 怎么把一句话说清楚 写一封信 措辞、格式、Few-shot
Context Engineering 怎么把必要信息喂进去 准备一份档案 文档组织、RAG
Harness Engineering 怎么把整个环境搭成系统 建一座工厂 约束、反馈、稳定性

LangChain 公式：Agent = Model + Harness

Harness 是除了模型本身之外的一切——system prompt、工具定义、执行沙箱、编排逻辑、hooks 和中间件。模型是引擎，Harness 是整辆车。

二、架构维度：三 Agent 协作模型

┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Planner   │────→│  Generator  │←───→│  Evaluator  │
│  (规划者)    │     │  (开发者)    │     │  (评估者)    │
└─────────────┘     └─────────────┘     └─────────────┘
       │                    │                    │
       ↓                    ↓                    ↓
   写 spec.md           写代码               评估报告
                      写 progress.md

Sprint Contract 机制： - Generator 和 Evaluator 在写代码前就"什么叫完成"达成一致 - 包含：用户故事 + 可验证验收标准（27 条示例）+ 可执行操作步骤 - Evaluator 用 Playwright 真实操作应用，精确到代码行号定位 bug

Agent 通信：没有 RPC、消息队列、共享内存，就是读写文件。

三、评估维度：对抗式评分体系

四维度加权（复刻 QoderWork 实战）： | 维度 | 权重 | 说明 | |------|------|------| | 功能完整性 | 40% | 核心功能是否全部实现 | | 交互一致性 | 25% | 操作流程是否与参照一致 | | 视觉保真度 | 20% | 界面还原度 | | 技术质量 | 15% | 代码质量、错误处理 |

评分校准： - 10 分 = 与参照完全不可区分 - 7 分 = 功能正常但有明显差异 - 4 分以下 = 基本不能用

严格要求：每个差异都要截图取证，报告里写的是 "按钮颜色是 #2563EB vs #3B82F6，border-radius 是 8px vs 4px" 这种级别，不接受"看起来不太一样"这种模糊描述。

终止条件： 1. 收敛：分数 ≥ 目标分数 2. 停滞：连续 2 轮无进步（stagnation） 3. 回退：分数下降，触发 revert

四、实践维度：企业级落地案例

案例 1：复刻 QoderWork（小红书实践）

背景：没有完备 PRD 和设计稿，无法人工描述每一个细节

三个指令： - /baseline：用 Playwright MCP 系统化探索参照产品，逐页截图、逐功能记录 - /evaluate：按 4 维度加权评分 - /eval-fix：GAN 风格对抗 loop，直到收敛或停滞

结果： | 轮次 | 分数 | 发现的问题 | |------|------|-----------| | 第一轮 | - | 各种基础问题 | | 第二轮 | 9.4/10 | JSON 流式输出"{"半括号泄露；空白 markdown 区块；聊天区被遮挡 | | 第三轮 | 9.8/10 | 代码区块深色主题与整体白色风格违和 → 改为浅色卡片 |

关键体会： - Evaluator 发现的问题和开发者看到的完全不同 - 流式渲染 bug 需要模拟真实用户操作才能捕获 - 基线采集的颗粒度决定评估上限

案例 2：企业级 Java 项目（桔了个仔实践）

三层约束体系：

第一层：架构级约束（Architecture as Harness） - Spring Boot MVC 分层：Controller → Service → Repository → Entity - 代码应该怎么放，结构清晰，对 AI 有指导作用

第二层：质量级约束（Quality as Harness） | 工具 | 作用 | |------|------| | Spotless | 自动格式化 | | Checkstyle | 编码规范 | | SpotBugs | 缺陷检测 | | JaCoCo | 90% 行覆盖率硬门槛 | | ESLint + Prettier | 前端一致性 |

第三层：流程级约束（Workflow as Harness） - Pre-commit hooks：lint + test + commit-msg 检查 - CI/CD Pipeline：PR 阶段 7 步检查，QA/RELEASE 阶段简化 - Skills 定义：本地开发流程、Git 工作流、PR 质量标准

方法论总结：五个核心原则

  1. 先了解模型的局限，再设计架构
    清楚 AI 在哪些特定环节容易出错，比选择什么流行框架更重要。

  2. 将隐性标准显性化，将上下文记忆文件化
    AI 的上下文状态是动态的，把关键信息落实到文件中更可靠。

  3. "建设"和"评审"尽量分开
    在工作流中引入独立的检查环节，保持客观的验证视角。

  4. 以实际运行结果为验收标准
    不要仅凭代码看起来"工整"就认为大功告成，"能跑通"才是真正的交付。

  5. 所有的方法论都是随着模型进化的
    模型在进步，工程架构也需要跟着精简和迭代。不要让过去的变通方案成为未来的包袱。

结论与启示

未来判断

  1. Harness 不会消失，但形态会演化
    编译器变得越来越强大，我们并没有丢掉 CI/CD。验证和约束是独立于工具能力的需求。

  2. 利润流向集成环节
    模型正在被商品化，而 model + harness 的集成才是难以复制的差异化。Anthropic 和 OpenAI 的护城河可能不只是模型好，而是模型和 harness 的集成做得好。

  3. 工程师角色转变
    从"写代码"转向"设计环境、指定意图、建立反馈回路"。Humans steer. Agents execute.

行动建议

对于前端/增长工程团队：

  1. 立即行动：建立三层约束
  2. 架构级：明确项目结构和分层规范
  3. 质量级：强制 lint + test + coverage 门槛

  4. 流程级：CI/CD + pre-commit hooks

  5. 中期目标：实现 Vibe Coding 体验

  6. 需求 → AI 开发 → 自动验证 → 人工 review → 合并

  7. 人工介入点：plan 确认、feature 验证、mental alignment

  8. 长期建设：培养"工程规则设计师"

  9. 不再是语法熟练度竞争
  10. 而是架构设计、约束定义、反馈回路设计能力

一句话总结

Agent 的重点不在模型，而在环境。

模型是发动机，Harness 是整辆车。引擎再好，没有刹车你也不敢上路。Harness 写得越好，消失得越快——而这正是它的价值所在。
